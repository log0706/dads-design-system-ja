# 実装ガイド

この文書は公式DADSの原文ではなく、対象リポジトリへ安全に導入するためのローカル独自ガイドです。

## 導入前に確認する

1. 対象リポジトリの `AGENTS.md` とプロジェクト固有ルールを読む。
2. フレームワーク、言語、ビルド、CSS戦略、React・Tailwind等のバージョンを確認する。
3. 既存デザインシステム、デザイントークン、コンポーネントAPI、アクセシビリティ方針を確認する。
4. 公式コードのライセンスと対象コミットを確認する。
5. 追加依存関係、バージョン変更、ビルド設定変更は事前に影響を整理する。勝手に更新しない。

## プレーンHTML／CSS／JavaScript

- 公式HTMLサンプルとStorybookから、意味的なHTML構造、状態、キーボード操作、フォーカス、レスポンシブ挙動を確認する。
- CSSを丸ごと持ち込む前に、対象のリセット、詳細度、カスケードレイヤー、命名規則との衝突を確認する。
- JavaScriptを必要とするコンポーネントは、初期化、イベント解除、複数配置、動的追加、エラー時の劣化を確認する。
- 見た目だけを再現せず、ラベル、説明、エラー関連付け、読み上げ、フォーカス順を保つ。

## React／TypeScript

- 公式Reactサンプルの現行README、対象コンポーネント、型定義、Storybookを確認する。
- 対象のReact、TypeScript、Tailwind、テスト環境との差を先に列挙する。
- 既存のフォーム管理、ルーティング、状態管理、i18n、バリデーションへ適合させる。
- コンポーネントを一括移植せず、利用頻度とリスクの低い単位から段階導入する。
- React 18を前提とした公式サンプルをReact 19等へ入れる場合は、型エラーと実行時挙動を実測する。

## Tailwind CSS

- 対象のTailwindメジャーバージョンを確認する。
- v3はプラグイン設定、v4はCSSからの読み込みという公式リポジトリの現行手順を参照する。
- 既存テーマとDADSトークンの衝突、生成CSS量、コンテンツ検出、ダークモード、プリフライトの影響を確認する。
- DADSテーマ導入をTailwind本体のメジャーアップグレードと同時に行わない。

## 既存デザインシステムと共存する

トークンを次の順で対応付ける。

1. 意味: 例として本文、境界、危険、成功、フォーカス
2. 用途: 背景、文字、アイコン、操作、状態
3. 値: 色、サイズ、余白、角、影

値が近いだけで統合せず、意味と状態が一致するか確認する。既存APIを維持し、内部トークンだけを段階的に対応付ける方法を優先する。

## 段階導入する

1. タイポグラフィ、色、余白等の基礎トークンを棚卸しする。
2. ボタン、入力、エラー等の基礎コンポーネントを候補にする。
3. 1画面または1フローで試行する。
4. アクセシビリティと回帰を検証する。
5. 差分、例外、保守責任をスタイルガイドへ記録する。
6. 検証後に対象範囲を広げる。

## テストとアクセシビリティ

- 自動テスト: 既存のlint、型検査、単体、統合、E2E、アクセシビリティ検査を実行する。
- キーボード: Tab、Shift+Tab、Enter、Space、Escape、矢印キー等を対象コンポーネント仕様に応じて確認する。
- フォーカス: 可視性、順序、モーダル内制御、復帰位置を確認する。
- フォーム: ラベル、必須、説明、入力目的、エラーの関連付けと修正方法を確認する。
- 表示: コントラスト、文字拡大、ズーム、リフロー、狭幅、長い日本語、状態の非色依存を確認する。
- 支援技術: 重要フローはスクリーンリーダー等で手動確認し、未実施なら明記する。

## 報告する

- 参照した公式ページまたは公式Markdown
- 公式コードを使用した箇所とライセンス
- DADSから変更した箇所と理由
- 追加・変更した依存関係
- 実行した検証と結果
- 未実施の検証、互換性リスク、今後の更新方法
